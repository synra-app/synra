# 主机选举原则

## 目标

在主机失效时，集群在可控时间内自动选出新主机并恢复中继能力。

## 触发条件

节点进入选举流程的条件：

- 主机主动发布下线声明（优先触发）。
- 连续多个心跳周期未收到主机心跳（被动检测兜底）。
- 会话层确认主机连接不可恢复。

## 成功标准

- 新主机在规定窗口内获得多数投票。
- 新主机发布主机公告并被多数成员确认。
- 成员在恢复窗口内完成重连并可继续路由请求。

## 选举安全原则

- 任期单调递增：任何新选举必须提升 `term`。
- 单任期单票：每个节点在同一 `term` 只投一次票。
- 多数派原则：仅多数票候选者可成为主机。
- 过期拒绝：低 `term` 消息一律拒绝处理。

## 冲突处理原则

- 同时出现多个候选者时，以更高 `term` 优先。
- 同 `term` 冲突时按确定性排序（如 `nodeId` 字典序）裁决。
- 旧主机恢复上线后不得直接夺回主机角色，必须以成员身份加入。

## 恢复原则

- 新主机接管后先恢复路由表，再开放新请求入口。
- 未完成请求按请求状态执行补偿：
  - 可重放请求进入幂等重放队列。
  - 不可重放请求返回 `HOST_FAILOVER_ABORTED`。
- 成员本地缓存主机信息，主机公告更新后原子切换。
- 主机下线后必须触发重新选主，且成员只在新主机公告生效后恢复业务写入。

## 下线通知原则

- 成员设备计划下线时，先向主机发送 `session.close` 主动通知。
- 主机收到成员下线通知后，向其余成员广播成员状态变更。
- 主机计划下线时，先广播 `host.retire`，随后进入主机切换窗口。

## 时序目标（建议）

- 心跳周期：10 秒一次（用于被动健康检测）。
- 主机失联检测：20~30 秒内触发（连续心跳缺失兜底）。
- 完成选举：1~3 秒内完成多数票确认。
- 完成接管：3~8 秒内恢复主链路可用。
