# 04 现代技术栈与性能策略

## 目标

在实现 `@synra/capacitor-electron` 时，优先采用现代且稳定的库组合，同时给出可执行的性能约束，避免“过度封装导致性能回退”。

## 技术栈建议

## 核心运行时

- **Electron**：使用当前稳定大版本（锁定到明确主版本，按月度评估升级）。
- **Capacitor**：与 monorepo 主线版本一致，避免插件层版本漂移。
- **Node.js**：对齐仓库工具链版本，避免 Electron 构建链路差异。

## 类型与协议

- **TypeScript**：严格模式，公开 API 全量导出类型。
- **zod（或同类 schema 库）**：跨进程 payload 运行时校验。
- **nanoid/uuid**：生成 requestId，支持追踪与排障。

## 日志与观测

- **结构化日志方案**：统一字段（requestId、method、durationMs、status）。
- **轻量性能埋点**：覆盖冷启动、首调用、高频调用三类路径。

## 测试体系

- **vite-plus/test**：单元与契约测试入口。
- **Playwright（可选）**：端到端验证 WebView + Electron 交互。
- **Mock Electron 环境**：隔离纯协议测试与宿主测试。

## 封装优雅性原则

## API 层

- 面向业务暴露少量高价值 API，禁止直接外泄 IPC 细节。
- API 命名遵循语义化，不暴露技术实现名词。

## Bridge 层

- 单一调用入口（统一埋点、错误处理、超时控制）。
- 中间件化处理流程：校验 -> 调用 -> 转换 -> 记录。

## Host 层

- service 以接口驱动，底层 Electron/Node 能力通过 adapter 注入。
- 每个 service 尽量无共享可变状态，必要状态集中托管。

## 性能预算（建议基线）

- 冷启动至桥接可用：`< 1500ms`（开发环境可放宽，生产需收紧）。
- 单次 IPC 往返（小 payload）：P95 `< 8ms`。
- 高频调用场景（1000 次轻量调用）：错误率 `0%`，且无明显内存爬升。
- 单次 payload 序列化大小：建议 `< 256KB`，超限走文件或流式方案。

## 性能优化策略

## 1) 减少不必要 IPC

- 可在渲染层本地完成的逻辑，不经过主进程。
- 批量请求优先设计为批处理 API，减少往返次数。

## 2) 控制序列化开销

- payload 只传必要字段，避免大对象整包传输。
- 二进制数据尽量走文件路径或共享资源引用，而非 base64 大字符串。

## 3) 主进程任务隔离

- CPU 密集任务避免阻塞主线程，必要时拆到 worker/子进程。
- 统一超时与取消策略，防止任务堆积。

## 4) 缓存策略

- 对稳定元信息（版本、环境信息）做短周期缓存。
- 缓存命中必须可观测，可配置关闭以便排障。

## 工程约束

- 新增 API 必须同步更新：类型、schema、错误码、测试。
- 新增 handler 必须有性能与错误场景测试。
- 任何绕过统一 invoke 管道的调用都视为设计违规。

## 升级策略

- Electron 升级先跑兼容清单：安全配置、API 弃用、打包行为。
- 协议版本升级必须保持至少一个旧版本兼容窗口。
- 重大变更通过 feature flag 灰度启用。
