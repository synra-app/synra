# 03 现代技术栈与性能策略

## 目标

定义 `@synra/capacitor-electron` 的工程化基线，确保从零实现后可以长期开源维护：

- 技术选型可复现、可升级、可跨平台验证。
- 性能预算可量化、可回归、可解释。
- 发布与兼容策略透明，便于社区集成与迁移。

## 技术栈基线

### 核心运行时

- Electron：锁定到明确主版本区间，按季度评估升级。
- Capacitor：对齐主仓库稳定版本，避免插件层版本漂移。
- Node.js：与仓库工具链一致（建议 LTS），确保构建行为可复现。
- TypeScript：strict 模式，公开 API 必须有类型导出。

### 协议与校验

- 使用 schema 库（如 zod）做请求与响应双向校验。
- 使用稳定 ID 方案（如 uuid/nanoid）生成 requestId。
- 协议定义与错误码定义统一放在 `src/shared`。

### 日志与可观测性

- 结构化日志字段最小集合：`requestId`、`method`、`durationMs`、`status`、`errorCode`。
- 性能埋点覆盖：冷启动、首个 API 调用、高频调用、失败调用。
- 默认不记录敏感 payload，必要时做脱敏后采样记录。

### 测试体系

- `vp test`：单元测试与契约测试入口。
- 集成测试：覆盖 `renderer -> preload -> main -> service` 主链路。
- E2E（可选）：使用 Playwright 等工具验证真实宿主行为。
- Mock 策略：协议测试可 mock Electron，能力测试需至少一条真实链路。

## 兼容矩阵策略

建议在仓库中维护显式矩阵（可放在 README 或单独文档），至少包含以下维度：

- OS：Windows、macOS、Linux。
- Electron：当前支持主版本与下限主版本。
- Capacitor：当前支持主版本。
- Node.js：开发与 CI 使用版本。

示例矩阵（占位）：

| OS     | Electron | Capacitor | Node.js | 状态      |
| ------ | -------- | --------- | ------- | --------- |
| win32  | 34.x     | 8.x       | 22.x    | supported |
| darwin | 34.x     | 8.x       | 22.x    | supported |
| linux  | 34.x     | 8.x       | 22.x    | supported |

## 版本与弃用策略

- 遵循 SemVer：破坏性变更仅在 major 发布。
- 每次 major 变更需提供迁移指南与替代方案。
- 保持至少一个 minor 兼容窗口（协议与 API）。
- 弃用流程：`deprecated` 标记 -> 文档声明 -> 下一个 major 删除。

## 性能预算（默认基线）

- 冷启动到桥接可用：P95 `< 1500ms`（生产环境）。
- 单次 IPC 往返（小 payload）：P95 `< 8ms`。
- 高频调用（1000 次轻量请求）：错误率 `0%`，无明显内存持续爬升。
- 单请求 payload 建议 `< 256KB`，超限改用文件或流方案。

## 性能优化策略

### 1) 减少 IPC 往返

- 优先在渲染层完成纯计算逻辑。
- 对批量操作提供 batch API，避免频繁小调用。

### 2) 控制序列化成本

- payload 只传必要字段，禁止传递整个上下文对象。
- 二进制数据优先路径引用或分片，不走大体积 base64。

### 3) 防止主进程阻塞

- CPU 密集任务移入 worker 或子进程。
- 所有长任务都必须支持超时和取消。

### 4) 明确缓存边界

- 仅缓存稳定元数据（runtimeInfo、capabilities）。
- 缓存必须可观测、可禁用、可失效。

## 工程约束

- 新增 API 必须同步更新：类型、schema、错误码、文档、测试。
- 新增 handler 必须包含失败路径和性能测试。
- 禁止绕过统一 invoke 管道直接新增桥接入口。
- PR 必须说明对兼容矩阵和安全基线的影响。

## 依赖升级策略

- Electron 升级前执行兼容检查：安全默认、API 弃用、打包行为。
- Capacitor 升级前执行契约回归：插件注册、调用语义、异常映射。
- 升级流程：预发布分支验证 -> 文档更新 -> 正式发布。

## CI 验证建议

- 每次 PR 运行：`vp check` + `vp test`。
- 发布前运行：跨 OS 集成测试矩阵（至少 win32 + darwin）。
- 若兼容矩阵变化，必须附带回归结果摘要。
