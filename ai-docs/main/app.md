# 起因

现在虽然各生态会有很方便的多设备互通场景，例如共享剪切板、快速分享文件

# 不足

受限于生态约束，功能相对比较单一

# 想法

做一个软件，分为手机端和 PC 端，两方都作为一个框架，通过引入插件的形式，实现具体功能

两种端都可以实现 **分享** 和 **接收分享**，做不同的处理

# 场景举例

假设我在手机看文章，发现一个 `Github` 地址或仓库名，我想用电脑打开查看

## 基于目前我自己的方式

我的手机电脑是同一个生态，电脑有相应软件，最快速的是手机直接复制，电脑端就会有剪切板记录

但实际情况大多都是，手机默认是没有和电脑连接的，你需要先连接才能复制内容

接下来打开新的浏览器标签，粘贴，如果只是仓库名你还需要手动拼接 `github.com`

## 基于本软件的设想

复制后通过打开手机软件，此时会提示读取到新的剪切板内容

或者复制后，会弹出分享，直接分享到该软件

匹配到 `Github`，弹出**是否用电脑浏览器打开地址**的确认弹窗，点击确认后，电脑打开对应网页

# 具体实现细节

其中 `分享` 部分，可以通过验证当前分享的内容，判断是否触发这个插件

假设有以下几种，手机端分享类型

- 普通文本
- 链接
- 文件

这时拿到链接或者文本后，会有一个需要插件实现的抽象方法，叫处理分享内容

调用后返回处理后的结果，来判断当前插件是否适用

比如我在验证分享中使用正则表达式匹配是否是 `Github` 链接，或者是仓库格式

`/(github\.com|^\w+\/\w+$)/`，这让以下两种都会匹配

```
imba97/smserialport
https://github.com/imba97/smserialport
```

接下来我再验证，如果是第一种，则手动补全成链接格式，最终返回

如果没有匹配到，则返回一个特定约束值，表示没有触发到当前插件

## 匹配电脑端方法

比如，电脑端有以下抽象方法

- 打开文件
- 用浏览器打开
- 用软件打开

此时我匹配到的是链接，所以会匹配到一个选项就是打开网页

如果匹配到多个，会让用户在手机上选择

## 执行电脑方法

把链接以及其他相关信息作为参数调用电脑端方法

电脑的软件会调用浏览器，打开 `Github`

## 插件可配置

比如无需确认弹窗，直接使用电脑打开

# 更多畅想

除了上面的分享功能，手机端完全可以变为一个快捷键

手机端也可以通过插件的形式，构造快捷键

例如有一个 **PS 快捷键** 插件，点击进入后会是一个页面

里面有各种功能，例如撤销、重做、魔棒、自由变换等

实际上只是手机点击功能键后，发送给电脑，电脑模拟按快捷键
