# 激进式重构

## 适用场景

- 用户明确提出“重构 / 重写 / 改造 / 架构优化”。
- 用户要求“清理冗余、删旧代码、去历史包袱”。
- 用户希望通过设计模式提升可维护性与性能。

## 核心原则

1. **激进式重构**：默认不保留向后兼容层，不引入过渡适配代码。
2. **冗余即删除**：发现重复实现、无效抽象、死代码、过时分支时，直接移除。
3. **优雅与性能并重**：优先选择结构清晰、职责单一、运行高效的方案。
4. **设计模式服务于问题**：按实际复杂度选用合适模式，避免过度设计。
5. **边界清晰**：模块职责必须可解释、可验证、可测试。
6. **工作区联动修改**：必须检查 `pnpm workspace` 中所有关联项目；只要与本次重构相关，必须同步修改，不允许仅改局部留下不一致状态。

## 强制边界（项目特定）

- `capacitor-lan-discovery` **仅**用于设备发现能力。
- `capacitor-device-connection` **仅**用于设备连接能力。
- 以上两个库**禁止增加任何业务逻辑**（例如业务状态编排、业务规则判断、业务流程聚合、页面/产品域策略）。
- 若发现已有业务逻辑混入这两个库，重构时应迁移到上层业务模块，并保持库职责纯净。

## 消息字段白名单（强制）

- 消息信封字段只允许：`requestId`、`event`、`target`、`from`、`replyRequestId`、`payload`、`timestamp`。
- 任何情况下都**禁止**擅自新增消息字段；若出现新增字段需求，必须先经用户明确确认后再调整。
- 不做兼容层：禁止新旧字段并存、双写、回退映射。

## 执行流程

1. 明确目标：定义本次重构要消除的复杂度与性能瓶颈。
2. 划清边界：先确定模块职责与依赖方向，再动代码。
3. 工作区扫描：按依赖关系与调用链检查 `pnpm workspace` 相关项目，定位所有受影响点。
4. 直接替换：用目标实现替换旧实现，不保留兼容桥接层。
5. 清理冗余：删除旧接口、废弃文件、重复逻辑与无用配置。
6. 模式落地：在必要处引入模式（如策略、工厂、组合、适配），并保持最小化。
7. 验证结果：确保行为正确、性能不退化、结构更简单，且关联项目保持一致。

## 输出要求

- 说明删除了哪些冗余与历史兼容层。
- 说明采用了哪些设计模式，以及为什么比原方案更优雅。
- 说明性能收益来源（复杂度、渲染/IO路径、对象创建、缓存策略等）。
- 列出 `pnpm workspace` 中本次联动修改的关联项目及修改原因。
- 明确确认：`capacitor-lan-discovery` 和 `capacitor-device-connection` 未承载业务逻辑。
