# Frontend UI Rules

适用于 `apps/frontend` 的页面、组件与样式实现规范。

## 适配策略

- 同一套前端覆盖桌面端、移动端和 Web；如涉及平台能力（如 Electron 窗口控制）可加最小化分支。
- 采用移动优先布局：先保证 `base/xs/sm` 可用，再增强 `md/lg/xl` 信息密度。
- 大屏适配只改变排布，不改变主流程顺序与关键动作位置。

## UnoCSS 断点与常见模式

- 使用 `uno.config.ts` 中断点：`xs/sm/md/lg/xl/2xl/3xl`。
- 常见容器模式：
  - 页面容器：`mx-auto w-full max-w-* px-*`
  - 双栏：`grid grid-cols-1 md:grid-cols-12`
  - 三栏：`lg:grid lg:grid-cols-12`
- 常见栅格模式：
  - `grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3`
- 禁忌：
  - 在页面中堆叠大量临时类导致语义不清
  - 断点跳级覆盖且无说明
  - 不同页面同一交互采用不同断点策略

## 自动导入与 composables

- 前端默认使用自动导入，减少手写 `import`。
- 自动导入范围至少包含：
  - `vue`
  - `vue-router`
  - `src/composables` 下 `useXxx`
- composable 规范：
  - 目录：`src/composables`
  - 命名：`useXxx`
  - 职责单一，按领域拆分（设备、消息、日志、执行态）

## 组件封装约定

- 页面负责流程编排，组件负责状态展示与事件发射。
- 领域组件不直接耦合路由与 store，实现通过 props/emits 交互。
- 同类展示块（卡片、会话列表、日志列表）优先复用组件，不重复粘贴模板。
- 全局交互按钮统一走 `AppButton`，避免页面内零散自定义按钮样式。

## 视觉与主题约束（暗色 + 毛玻璃）

- 默认主题为暗色，不新增浅色专用分支；颜色使用语义 token，不在业务组件里硬编码临时色值。
- 毛玻璃效果通过统一语义类复用（如 `glass-panel`、`glass-panel-soft`、`glass-button`），禁止每个页面单独拼接 blur/border/shadow。
- 边框从简：避免“一层套一层”的边框堆叠。优先使用层级对比（透明度、阴影、背景亮度差）表达结构。
- Topbar 固定在最上方，不参与内容滚动；下方 container 内的侧栏与主内容区独立滚动。

## 按钮与交互反馈约束

- 按钮默认采用 ghost 原则：非主操作在静止态不显示实底，hover/focus 才显现半透明毛玻璃反馈。
- 主操作可使用实底或增强态（如 `solid`），危险操作使用 `danger`，但仍保持统一圆角、动效、焦点环。
- 禁止出现同一页面中多套按钮交互风格（比如部分按钮常驻背景、部分 hover 才显示且规则不一致）。
- 所有可交互元素必须有明确 focus-visible 样式，不得移除键盘可见反馈。

## Hover 原则

### 1) 滚动条 Hover 原则

- 滚动条只在用户 hover 到对应滚动容器时出现，未 hover 时隐藏。
- 滑槽（track）保持透明，仅展示滚动 thumb，避免遮挡与视觉噪音。
- 滚动容器需预留右侧安全空间，不能让滚动条覆盖正文内容。
- 该规则仅作用于指定滚动容器（如 `.app-scroll-container`），不要全局覆盖所有元素滚动条。

### 2) 非重要操作 Hover 原则

- 非关键操作优先弱化展示，hover/focus 时再增强反馈（图标、次级按钮、快捷动作）。
- 该规则按场景逐步落地，不要求一次性改完全部历史页面。
- 重要主流程操作不适用该规则，必须始终可见。

## 验收要求

- 小屏可完整执行主流程，关键操作无需横向滚动。
- 中屏与大屏布局提升信息效率，但不增加认知路径。
- 自动导入场景通过类型检查与 lint，无未定义误报。
- 新页面遵循统一术语和状态文案。
- 新增交互遵循「暗色 + 毛玻璃 + ghost hover」统一规则，避免回退到旧样式。
