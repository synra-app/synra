# Coding Rules (TS / Node / Vue)

## 通用

- 优先保证可读性与类型安全，避免技巧化写法。
- 新增或修改逻辑时，优先补充对应测试。
- 仅在必要时加注释，注释解释“为什么”，不解释显而易见的“做什么”。

## 语言与文本约束

- 代码、单元测试、代码注释统一使用英文。
- 文档（如 `*.md`）可使用中文。
- 用户明确要求保留的业务文案可例外（如测试输入的固定 label）。

## TypeScript / Node

- 保持 ESM 风格导入导出。
- 使用严格类型，避免 `any`；如必须使用，说明原因并缩小影响范围。
- 优先通过导出的类型与接口建模跨包边界，不把内部实现细节泄漏到外部。

## Vue

- 优先使用 `script setup` 与组合式 API。
- 页面逻辑与展示分离，复杂状态抽离到 composable/store。
- 保持模板语义清晰，避免在模板中塞入复杂表达式。

## 自动导入（前端）

- 在 `apps/frontend` 中优先采用 `unplugin-auto-import`，减少样板 `import`。
- 默认自动导入来源应覆盖：
  - `vue` 常用组合式 API
  - `vue-router` 常用路由 API
  - `src/composables` 下 `useXxx` 导出
- 允许显式 `import` 的例外：
  - 命名冲突导致可读性下降
  - 第三方非常规 API 需强调来源
  - 临时过渡阶段且已在代码评审说明

## UnoCSS 规则（前端）

- 使用 `apps/frontend/uno.config.ts` 定义的断点（`xs` 到 `3xl`）。
- 响应式写法采用移动优先：先写基础类，再用 `sm:`、`md:`、`lg:` 等逐步覆盖。
- 页面布局优先复用稳定模板（单列、双栏、三栏），避免临时拼凑断点类。
- 重复出现的类组合应抽成组件结构或约定模式，不在多个页面重复粘贴。

## 工具链

- 使用 `vp` 命令，不直接使用 `pnpm`/`npm` 执行同类任务。
- 提交前至少通过 `vp lint`；必要时补跑 `vp test` 与 `vp check`。
