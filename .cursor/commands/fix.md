# /fix

用于快速修复当前问题（lint、类型、测试失败或明显 bug）。

## 执行步骤

1. 先稳定复现问题，明确触发条件、输入、环境与预期行为。
2. 做完整链路分析（入口 -> 核心处理 -> 外部依赖 -> 状态落点），定位根因，不只处理表象。
3. 优先做根因修复，避免“打补丁式”临时规避；不顺带做无关重构。
4. 若根因暂时无法确定：在不确定节点增加可观测日志（关键入参、分支、错误码、上下文 ID），并说明如何触发与采集。
5. 引导用户按步骤复现并回传日志；基于日志收敛可疑范围后再完成修复。
6. 对改动文件执行检查（至少 `vp lint`，必要时 `vp test` / `vp check`）。
7. 输出修复说明：问题链路、根因、改动点、验证结果、剩余风险与后续观察项。

## 约束

- 遵守仓库规则：使用 `vp`，不要直接用 `pnpm` 执行等价命令。
- 代码/测试/注释使用英文；文档可中文。

## 日志埋点要求（根因未明时）

- 日志要能回答“数据从哪里来、经过了什么分支、在哪里偏离预期”。
- 同一轮问题排查的临时日志统一使用同一个 tag，格式为 `[xxx]`（示例：`[discovery-debug]`），便于全链路检索。
- 每条日志包含最小必要上下文：请求/会话 ID、设备 ID、关键参数快照、时间戳。
- 避免噪音和敏感信息泄露：不打印密钥、完整凭据、超大对象。
- 给用户明确回传指引：执行命令、复现步骤、日志文件位置与筛选关键词。
- 问题修复并验证通过后，移除本次排查新增的临时日志，避免长期污染日志流。
